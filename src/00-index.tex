\section{Canonical Product Definition}

Flytics will build a secure marketplace and operations platform for drone-based client services. Clients request missions and analytics through the web portal, qualified contracted pilots bid on or receive assignments, internal operations teams govern delivery, and clients receive data products, reports, invoices, and service history through a controlled account experience.

The platform includes an AI Services Layer, but the production system must not depend on AI as an uncontrolled decision maker. AI capabilities provide recommendations, extraction, ranking, summarization, and analysis support. Final operational controls, auditability, override paths, and deterministic business rules remain part of the core platform.

\section{Canonical Core Engine Scope}

The core engine is the shared backend domain layer used by the web portal, pilot portal, operations console, AI services, analytics pipeline, billing module, and integrations. It owns the following capabilities:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Request and mission model:} canonical records for client request, mission, site, schedule window, service package, deliverables, urgency, attachments, and status.
    \item \textbf{Eligibility and compliance:} pilot qualification checks, certification and insurance validity, equipment fit, location coverage, client restrictions, and job release rules.
    \item \textbf{Pricing and bidding:} baseline price calculation, configurable price corridors, pilot bid intake, bid comparison, platform fee calculation, and approval workflow.
    \item \textbf{Matching and assignment:} deterministic eligibility filtering plus ranked recommendations from proximity, availability, performance, equipment, cost, and risk inputs.
    \item \textbf{Workflow orchestration:} state transitions from draft request through intake, quote, bidding, assignment, flight completion, analytics, client approval, invoicing, and closure.
    \item \textbf{Analytics orchestration:} upload registration, file metadata, processing jobs, model outputs, quality review, report packaging, and deliverable publishing.
    \item \textbf{Frame store architecture:} frame-level records per drone, expandable frame chains, frame-to-frame lineage, and per-operation frame grouping.
    \item \textbf{Operation stack switching:} deployed stack profiles that can be switched per operation while preserving deterministic state and audit history.
    \item \textbf{Distributed task execution:} Apache Spark-based task runner for queued analytics and transformation tasks with retry and traceability.
    \item \textbf{Billing events:} invoice-ready events, payment status, pilot payout records, subscription events, refunds, credits, and revenue attribution.
    \item \textbf{Audit and observability:} immutable business event log, user and system actions, AI recommendation inputs and outputs, override reasons, metrics, traces, and alerts.
    \item \textbf{Integration contracts:} stable API boundaries for identity, payments, storage, weather, airspace, notifications, mapping, AI providers, and accounting systems.
\end{enumerate}

\section{Shared Delivery Principles}

\begin{itemize}[leftmargin=*]
    \item Build the marketplace workflow first, then layer AI recommendations onto observable and testable business processes.
    \item Keep client, pilot, operations, finance, and support workflows separated by role-based access control.
    \item Treat compliance, audit history, data privacy, and override capability as launch requirements, not later enhancements.
    \item Use structured APIs and domain events for all cross-service communication that affects mission state, pricing, assignment, billing, or deliverables.
    \item Validate revenue assumptions with real demand, pilot supply, pricing, and margin data before committing to scale targets.
\end{itemize}
